# Lessons Learned

## 做对的事

- 把源站、Cloudflare、订阅客户端分层测试，最终定位到 Cloudflare Challenge，而不是反复重装 Xray。
- 为订阅新增标准 443 入口，绕开高端口 TLS 的客户端兼容性问题。
- WAF Skip 只覆盖精确 host 和两个随机订阅 path，没有关闭整个站点安全。
- DNS 只发布 A，避免未完成的 IPv6 路径引入随机失败。
- 保留 Reality 直连与 WS/TLS CDN 两类节点，主备故障面不同。
- 最终用出口 IP/ASN/地区确认流量确实来自新服务器，而不是只看节点名称。
- 用独立 mihomo 内核验证 Reality/WS，而没有再次切换承载控制通道的 Clash。
- 在最终 YAML 中发现 Reality XHTTP 错用了橙云域名，并改为直连地址；这一步不能被“节点数量正确”替代。

## 代价较高的错误

### 1. 在远程控制过程中切换未验证节点

结果是网络立即中断，代理失去继续修复自己的通道。以后必须用第二设备或独立测试内核；最终切换由用户执行。

### 2. 把“订阅下载失败”误判成“v2ray-agent 没装全”

订阅、节点握手和出口是不同层。节点数量少不是订阅 EOF 的证据。先对照源站 200 与 Cloudflare 响应头。

### 3. 认为美国 IP 理论上就能访问 Reddit

网站风控看 IP 信誉、ASN、自动化特征、Cookie 和协议路径。国家只是一项信号；IPv4/IPv6 应分别测试。

### 4. 同时改变太多客户端变量

同时启用 TUN、系统代理、IPv6、QUIC 和新 profile，会让错误无法归因。每次只改一个变量，并保留快速回滚。

### 5. 直接暴露上游高端口订阅

某些 Windows/Android TLS 栈在非标准端口上出现 EOF 或协商差异。通过标准 443、正确证书和精确 Nginx 路由更稳定。

### 6. 把 provider 直链交给客户端当完整配置

`/s/clashMeta/<TOKEN>` 只有节点，没有代理组和规则。全局模式可以很快，甚至 YouTube 4K 流畅，
但这不能证明规则模式正确。客户端直接导入必须使用 `/s/clashMetaProfiles/<TOKEN>`，验证器也必须按
端点用途检查结构，而不是看到任意 YAML marker 就通过。

### 7. 把橙云/灰云切换当作通用排错

同一主机同时承担订阅/WS 443 和 Reality/XHTTP 高端口时，切橙云会修好一边、破坏另一边。正确做法
是分离 `EDGE_HOST` 与 `DIRECT_HOST`，或让直连节点使用源站 IP。公共 DNS 生效后再测试，不以控制台
按钮状态代替解析结果。

### 8. 只看 TCP 端口可达

端口可达但 Xray inbound 没有正确响应时，客户端仍会全部超时。必须用协议内核完成握手和 HTTPS
请求；若多个客户端、多个协议同时超时，优先检查/重启单个 Xray 服务，而不是继续改手机设置。

### 9. 把两项独立测试写成“全部可用”

独立测试的结论必须限定到实际节点。Reality Vision 和 Reality XHTTP 返回 `204`，只能证明这两项
通过；手机规则、其他节点、IPv6、QUIC 和特定网站仍要各自验证。

## 后续默认策略

- 最少协议、全部验证：Reality 主 + WS/TLS 备，兼容性确需时才加 VMess。
- IPv4 优先；IPv6 不是“越多越好”，而是需要完整验证的另一条网络。
- 先源站、再 Cloudflare、后客户端。
- 先验证完整 profile/provider 的用途和结构，再交付任何“直链”。
- 每次上游重新生成订阅后重跑节点路由审计，防止 XHTTP 再次继承 CDN 地址。
- 先验证、后切换；先重启复测、后清理旧资源。
- 所有自动化都要有备份、门禁和可观察证据。
