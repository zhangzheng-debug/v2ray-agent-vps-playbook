# Lessons Learned

## 做对的事

- 把源站、Cloudflare、订阅客户端分层测试，最终定位到 Cloudflare Challenge，而不是反复重装 Xray。
- 为订阅新增标准 443 入口，绕开高端口 TLS 的客户端兼容性问题。
- WAF Skip 只覆盖精确 host 和两个随机订阅 path，没有关闭整个站点安全。
- DNS 只发布 A，避免未完成的 IPv6 路径引入随机失败。
- 保留 Reality 直连与 WS/TLS CDN 两类节点，主备故障面不同。
- 最终用出口 IP/ASN/地区确认流量确实来自新服务器，而不是只看节点名称。

## 代价较高的错误

### 1. 在远程控制过程中切换未验证节点

结果是网络立即中断，代理失去继续修复自己的通道。以后必须用第二设备或独立测试内核；最终切换由用户执行。

### 2. 把“订阅下载失败”误判成“v2ray-agent 没装全”

订阅、节点握手和出口是不同层。节点数量少不是订阅 EOF 的证据。先对照源站 200 与 Cloudflare 响应头。

### 3. 认为美国 IP 理论上就能访问 Reddit

网站风控看 IP 信誉、ASN、自动化特征、Cookie 和协议路径。国家只是一项信号；IPv4/IPv6 应分别测试。

### 4. 同时改变太多客户端变量

同时启用 TUN、系统代理、IPv6、QUIC 和新 profile，会让错误无法归因。每次只改一个变量，并保留快速回滚。

### 5. 直接暴露上游高端口订阅

某些 Windows/Android TLS 栈在非标准端口上出现 EOF 或协商差异。通过标准 443、正确证书和精确 Nginx 路由更稳定。

## 后续默认策略

- 最少协议、全部验证：Reality 主 + WS/TLS 备，兼容性确需时才加 VMess。
- IPv4 优先；IPv6 不是“越多越好”，而是需要完整验证的另一条网络。
- 先源站、再 Cloudflare、后客户端。
- 先验证、后切换；先重启复测、后清理旧资源。
- 所有自动化都要有备份、门禁和可观察证据。
